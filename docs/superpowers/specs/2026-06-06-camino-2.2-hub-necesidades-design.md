# Camino 2.2 — Hub de necesidades

**Fecha:** 2026-06-06
**Repos involucrados:** `academconnect` (backend), `academconnect-web` (frontend)

## Contexto

Tras Camino 1 (importación legacy del admin) y Camino 2.1 (estudiante crea borrador
e invita orientador), Camino 2.2 cierra el TCC value flow inverso: el profesor publica
una necesidad concreta (trabajo en estado `ABIERTO`) y los estudiantes pueden
postularse vía `SolicitudVinculacion`. Al aceptar, el trabajo queda vinculado y
transiciona a `EN_DESARROLLO`.

La entidad `SolicitudVinculacion` y el service ya existen — este camino agrega lo que
falta: publicación con TTL configurable, cierre manual, cancelación por estudiante,
endpoints agregados (mis solicitudes, recibidas) y todo el frontend.

## Objetivos

- El profesor puede publicar un `BORRADOR` propio como `ABIERTO` eligiendo duración (7/15/30/60 días).
- El sistema cierra automáticamente cada `ABIERTO` cuya `expira_en` pasó.
- El profesor puede cerrar manualmente una publicación `ABIERTO` antes del TTL.
- El estudiante encuentra trabajos `ABIERTO` en un hub con filtros y ranking client-side.
- El estudiante puede postularse (`SolicitudVinculacion`) y cancelar la solicitud mientras esté `PENDIENTE`.
- El profesor ve todas las solicitudes recibidas a través de sus trabajos publicados.
- Al aceptar una solicitud, el trabajo pasa a `EN_DESARROLLO` con el estudiante asignado; las demás solicitudes pendientes sobre ese trabajo se auto-rechazan.

## No-objetivos

- Algoritmo de recomendación server-side (TF-IDF, embeddings, ELO de evaluadores).
- Notificaciones push o email a las partes.
- Negociación con coorientadores o equipos.
- Renovación automática de TTL — el profesor cierra y vuelve a publicar si lo necesita.

### Cambios en enums

- `EstadoSolicitud.CANCELADA` ya existe en el enum y en el CHECK SQL de
  `solicitud_vinculacion` (verificado en V4) — no requiere migración ni cambio de enum.
- `TipoActividad` suma:
  - `TRABAJO_PUBLICADO` — emitido en publicar.
  - `TRABAJO_CERRADO` — emitido en cierre manual.
  - `TRABAJO_EXPIRADO` — emitido por el job de TTL.
  - `SOLICITUD_VINCULACION_CANCELADA` — emitido cuando el estudiante cancela.

## API REST — backend

### Trabajos del profesor (`TrabajoController`)

Dos endpoints nuevos sobre el controller existente:

| Método | Path | Rol | Body | Descripción |

### Discovery (sin cambios)

El existente `GET /api/trabajos/buscar?estado=ABIERTO` es suficiente para el hub.
Cuando el caller está autenticado el parámetro `estado` se respeta tal cual; el flag
`soloPublicos` solo aplica al caller anónimo y hace `estado = APROBADO` forzado.

### Job de TTL

Nueva clase `TrabajoExpiracionJob` con un método `expirarVencidos()`:

```java
@Scheduled(cron = "0 0 3 * * *")  // 3am UTC diario
@Transactional
public void expirarVencidos() {
    Instant now = Instant.now();
    var vencidos = trabajoRepository.findByEstadoAndExpiraEnBefore(EstadoTrabajo.ABIERTO, now);
    for (var t : vencidos) {
        t.setEstado(EstadoTrabajo.CANCELADO);
        autoRechazarPendientes(t, "Trabajo expirado");
        events.publishEvent(ActividadEvent.of(
            TipoActividad.TRABAJO_EXPIRADO, null, "TRABAJO", t.getId(),
            Map.of("titulo", t.getTitulo()),
            VisibilidadActividad.PARTICIPANTES,
            participantes));
    }
}
```

Habilitar `@EnableScheduling` en una `SchedulingConfig` si todavía no existe en la app.

El método `expirarVencidos()` debe ser invocable también desde un test sin esperar al cron.

### Side-effect compartido: auto-rechazar pendientes

La lógica de "marcar `RECHAZADA` todas las solicitudes `PENDIENTE` de un trabajo con
una `respuesta` dada" se invoca desde tres lugares: `aceptar` (existente, ya lo hace
con `"Posición ocupada"`), `cerrar` (`"Trabajo cerrado"`), `expirarVencidos`
(`"Trabajo expirado"`). Extraer a un helper compartido en `SolicitudVinculacionService`
o `TrabajoService`:

```java
void autoRechazarPendientes(Trabajo trabajo, String respuesta) {
    var pendientes = solicitudRepository.findByTrabajoIdAndEstado(trabajo.getId(), EstadoSolicitud.PENDIENTE);
    Instant now = Instant.now();
    for (var s : pendientes) {
        s.setEstado(EstadoSolicitud.RECHAZADA);
        s.setRespuesta(respuesta);
        s.setResueltaEn(now);
    }
    solicitudRepository.saveAll(pendientes);
}
```

## Ranking del hub

Client-side. Una pequeña función pura recibe `trabajo` y `perfilDelUsuario` y retorna
un score `int`:

```ts
function scoreTrabajo(trabajo: TrabajoListItem, perfil: Perfil | null): number {
  if (!perfil) return 0;
  const perfilAreaIds = new Set(perfil.areas.map(a => a.areaId));
  const trabajoAreaIds = new Set((trabajo.areas ?? []).map(a => a.id));
  let intersect = 0;
  trabajoAreaIds.forEach(id => { if (perfilAreaIds.has(id)) intersect++; });
  return intersect * 10
    + lexicalMatch(trabajo.keywords ?? [], perfil.areas.map(a => a.areaNombre));
}
```

Donde `lexicalMatch` cuenta matches case-insensitive entre cada keyword y los nombres
de áreas del estudiante. Empates se rompen por `createdAt` desc.

Sin endpoint nuevo. Algoritmo más sofisticado queda para iteraciones futuras.

## Riesgos y decisiones abiertas

- **Notificaciones:** el profesor descubre solicitudes recibidas entrando a la app, y
  el estudiante descubre respuestas igual. Notificaciones reales (email/push) quedan
  fuera de scope.
- **Concurrencia en `aceptar`:** la lógica existente ya marca las demás solicitudes
  como `RECHAZADA` en la misma transacción, pero si dos profesores aceptan
  simultáneamente sobre el mismo trabajo, el segundo verá la validación
  `trabajo.estudiante != null` y lanzará `BusinessException`. Es aceptable.
- **Cancelación encadenada:** cuando el job de TTL expira un trabajo, las solicitudes
  PENDIENTE se auto-rechazan con respuesta `"Trabajo expirado"`. No se distinguen del
  rechazo manual a nivel de estado, pero la `respuesta` permite inferirlo en la UI.
- **El profesor puede "despublicar"?** Por simplicidad no — solo cerrar (`ABIERTO →
  CANCELADO`). Si quiere volver a publicar, crea un trabajo nuevo. Mantenerlo así
  evita reset de `expira_en` y conflictos con solicitudes ya recibidas.
- **Áreas y keywords obligatorias para publicar:** la validación al publicar exige
  ambos (insumos para discovery). El form de creación los pide opcionales, pero la
  publicación se bloquea hasta que estén completos. La UI debe guiar al usuario.
