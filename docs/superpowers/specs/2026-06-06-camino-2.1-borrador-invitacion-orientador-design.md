# Camino 2.1 — Borrador del estudiante + invitación a orientador

**Fecha:** 2026-06-06
**Repos involucrados:** `academconnect` (backend), `academconnect-web` (frontend)

## Contexto

Tras Camino 1 (importación de trabajos legacy por administrador), Camino 2.1 habilita al
estudiante a crear sus propios trabajos en estado `BORRADOR` e invitar formalmente a un
profesor para que sea su orientador. La aceptación del profesor vincula al trabajo y lo
transiciona a `EN_DESARROLLO`.

Este camino no incluye el hub de necesidades publicadas por profesores (`ABIERTO` con
`SolicitudVinculacion`), que queda para Camino 2.2.

## Objetivos

- El estudiante puede crear un trabajo propio en `BORRADOR` sin orientador.
- Puede editar el borrador mientras esté en `BORRADOR`.
- Puede invitar a un profesor específico a ser su orientador.
- El profesor invitado recibe la invitación y puede aceptarla o rechazarla.
- Si acepta, el trabajo queda vinculado y pasa a `EN_DESARROLLO` automáticamente.
- Una sola invitación pendiente por trabajo a la vez (modelo secuencial).

## No-objetivos

- Algoritmo robusto de recomendación de profesores (ELO, cargas, calificaciones) — Camino 2.2/futuro.
- Necesidades publicadas por profesor con estudiantes respondiendo — Camino 2.2.
- Mensajería bidireccional fuera del campo `motivo`/`respuesta`.
- Validación de capacidad o tope de orientación del profesor.

## Modelo de datos

### Nueva entidad: `InvitacionOrientacion`

| Campo | Tipo | Constraints |
|---|---|---|
| `id` | `bigserial` | PK |
| `trabajo_id` | `bigint` | FK `trabajo(id)`, NOT NULL |
| `profesor_id` | `bigint` | FK `profesor(id)`, NOT NULL |
| `estado` | `varchar` | NOT NULL, enum |
| `motivo` | `text` | nullable (mensaje del estudiante) |
| `respuesta` | `text` | nullable (mensaje del profesor) |
| `resuelta_en` | `timestamptz` | nullable hasta resolución |
| `created_at` | `timestamptz` | NOT NULL |
| `updated_at` | `timestamptz` | NOT NULL |

**Constraint clave:** partial unique index
`CREATE UNIQUE INDEX uq_invitacion_pendiente_por_trabajo ON invitacion_orientacion (trabajo_id) WHERE estado = 'PENDIENTE';`

Garantiza una sola invitación pendiente por trabajo, en línea con la decisión de cardinalidad
secuencial.

### Nuevo enum: `EstadoInvitacion`

```
PENDIENTE | ACEPTADA | RECHAZADA | CANCELADA
```

### Nuevos valores en `TipoActividad`

- `INVITACION_ORIENTACION_ENVIADA`
- `INVITACION_ORIENTACION_ACEPTADA`
- `INVITACION_ORIENTACION_RECHAZADA`
- `INVITACION_ORIENTACION_CANCELADA`

## API REST — backend

### Trabajos del estudiante

Nuevo controller `MeTrabajoController` con prefijo `/api/me/trabajos`. Todos los endpoints
exigen `hasRole('ESTUDIANTE')`.

| Método | Path | Body | Descripción |
|---|---|---|---|
| `POST` | `/api/me/trabajos` | `TrabajoEstudianteRequest` | Crea BORRADOR sin orientador, asigna `estudiante = currentUser`. Publica `TRABAJO_CREADO`. |
| `PUT` | `/api/me/trabajos/{id}` | `TrabajoEstudianteRequest` | Edita borrador. Valida ownership + `estado == BORRADOR`. |
| `GET` | `/api/me/trabajos` | — | Lista trabajos donde `estudianteId == currentUser`. |
| `GET` | `/api/me/trabajos/{id}` | — | Detalle del trabajo propio. |

**DTO `TrabajoEstudianteRequest`:**
```java
public record TrabajoEstudianteRequest(
    @NotBlank @Size(max = 300) String titulo,
    String descripcion,
    @NotNull TipoTrabajo tipo,
    Set<Long> areaIds,                 // opcional
    @NotNull @Size(min = 3, max = 8) List<@NotBlank @Size(min = 2, max = 50) String> keywords
) {}
```

### Invitaciones de orientación

Nuevo controller `InvitacionOrientacionController` con prefijo `/api/invitaciones-orientacion`.

| Método | Path | Rol | Body | Descripción |
|---|---|---|---|---|
| `POST` | `/api/invitaciones-orientacion` | `ESTUDIANTE` | `InvitacionOrientacionRequest` | Crea PENDIENTE. Validaciones abajo. |
| `POST` | `/api/invitaciones-orientacion/{id}/aceptar` | `PROFESOR` invitado | `RespuestaInvitacionRequest?` | ACEPTADA + side-effects. |
| `POST` | `/api/invitaciones-orientacion/{id}/rechazar` | `PROFESOR` invitado | `RespuestaInvitacionRequest?` | RECHAZADA. |
| `POST` | `/api/invitaciones-orientacion/{id}/cancelar` | `ESTUDIANTE` dueño | — | CANCELADA. |
| `GET`  | `/api/invitaciones-orientacion?estado=PENDIENTE` | `PROFESOR` | — | Recibidas por currentUser. |
| `GET`  | `/api/trabajos/{id}/invitaciones` | autenticado, dueño o invitado | — | Por trabajo. |

**Validaciones del `crear`:**
1. `currentUser` es ESTUDIANTE.
2. `trabajo.estudianteId == currentUser.id` (dueño).
3. `trabajo.estado == BORRADOR`.
4. `trabajo.orientadorId == null` (sin orientador).
5. No existe otra invitación con `(trabajoId, estado=PENDIENTE)`.
6. `profesorId` existe y `activo == true`.

**Side-effects de `aceptar`:**
- `invitacion.estado = ACEPTADA`, `respuesta = body.respuesta?`, `resuelta_en = now()`.
- `trabajo.orientador = profesor`, `trabajo.estado = EN_DESARROLLO`.
- Publica `INVITACION_ORIENTACION_ACEPTADA` con visibilidad `PARTICIPANTES` (estudiante + profesor).

**DTOs:**
```java
public record InvitacionOrientacionRequest(
    @NotNull Long trabajoId,
    @NotNull Long profesorId,
    @Size(max = 1000) String motivo) {}

public record RespuestaInvitacionRequest(@Size(max = 1000) String respuesta) {}

public record InvitacionOrientacionResponse(
    Long id, Long trabajoId, String trabajoTitulo,
    Long profesorId, String profesorNombre,
    EstadoInvitacion estado, String motivo, String respuesta,
    Instant resueltaEn, Instant createdAt) {}
```

### Seguridad

- Las validaciones de "dueño del trabajo" y "profesor invitado" se hacen en el service
  comparando `currentUser` (vía `SecurityContextHolder`) con los `id` del recurso.
- `@PreAuthorize` se queda con la verificación de rol; el ownership es semántica de dominio.

## Frontend

### Feature nueva: `features/mis-trabajos`

Lazy routes registradas dentro del shell con `authGuard`:

| Path | Guard adicional | Componente |
|---|---|---|
| `/mis-trabajos` | `roleGuard` con `ESTUDIANTE` | `MisTrabajosListPage` |
| `/mis-trabajos/nuevo` | `roleGuard` con `ESTUDIANTE` | `MisTrabajosCrearPage` |
| `/mis-trabajos/:id` | `roleGuard` con `ESTUDIANTE` | `MisTrabajosDetallePage` |

**`MisTrabajosListPage`:**
- Llama `GET /api/me/trabajos`.
- Cards o tabla con título, tipo, estado, fecha. Click → detalle.
- Botón "Crear trabajo" → `/mis-trabajos/nuevo`.

**`MisTrabajosCrearPage`:**
- Formulario reactivo: título (req), descripción, tipo (select), áreas (chips), keywords (chip-input 3-8).
- Sin orientador.
- Al submit → `POST /api/me/trabajos` → redirect a `/mis-trabajos/{id}`.

**`MisTrabajosDetallePage`:**
- Llama `GET /api/me/trabajos/{id}` y `GET /api/trabajos/{id}/invitaciones`.
- Render condicional según estado + presencia de invitación pendiente:
  - **BORRADOR + sin invitación pendiente:** muestra form editable (`PUT`) + panel "Invitar orientador" (componente `InvitarOrientadorForm`).
  - **BORRADOR + invitación PENDIENTE:** panel "Esperando respuesta de {profesor}" con botón "Cancelar invitación".
  - **EN_DESARROLLO o posterior:** vista read-only con orientador asignado y historial de invitaciones.

**`InvitarOrientadorForm` (component reutilizable):**
- Trae `GET /api/profesores` y los ordena client-side:
  1. Por overlap entre `profesor.areas[].areaId` y `trabajo.areas[].id` (si el trabajo tiene áreas).
  2. Fallback: por match léxico simple de `trabajo.keywords` con `profesor.areas[].areaNombre`.
- Selector de profesor + textarea opcional para `motivo`.
- Submit → `POST /api/invitaciones-orientacion`.

### Feature nueva: `features/invitaciones`

| Path | Guard adicional | Componente |
|---|---|---|
| `/invitaciones-orientacion` | `roleGuard` con `PROFESOR`, `EXTERNO` | `InvitacionesRecibidasPage` |

- Llama `GET /api/invitaciones-orientacion?estado=PENDIENTE`.
- Lista con título del trabajo, estudiante, motivo, fecha.
- Por item PENDIENTE: botones "Ver detalle" (link a `/repositorio/{trabajoId}`) + "Aceptar" + "Rechazar" (modal con textarea opcional).
- Tabs por estado (PENDIENTE / HISTÓRICO).

### Servicios frontend

- `MisTrabajosService`: `listar()`, `crear()`, `actualizar()`, `getById()` → `/api/me/trabajos`.
- `InvitacionOrientacionService`: `enviar()`, `aceptar()`, `rechazar()`, `cancelar()`, `listarRecibidas()`, `listarPorTrabajo()` → `/api/invitaciones-orientacion`.

### Backend — organización de service

Dos opciones equivalentes; el implementador escoge según la convención del repo:
- Agregar métodos `crearPorEstudiante`, `actualizarBorradorPorEstudiante`, `listarPropios` a
  `TrabajoService` (manteniendo el bounded context "trabajo" en un solo service).
- Crear `MeTrabajoService` separado para aislar las operaciones del estudiante sobre sus
  trabajos.

La preferencia inicial es **ampliar `TrabajoService`** porque la lógica comparte
`normalizarKeywords`, mapping y eventos.

### Sidebar

- `SECTIONS_ESTUDIANTE`: el item "Mis trabajos" pasa de placeholder a `route: '/mis-trabajos'`.
- `SECTIONS_EVALUADOR`: agregar "Invitaciones de orientación" → `/invitaciones-orientacion`.

## Recomendación de profesores

Para este camino mantenemos un ranking **client-side simple**:

1. Si el borrador tiene áreas asignadas: ordenar profesores por tamaño de la intersección entre
   `trabajo.areas[].id` y `profesor.areas[].areaId`.
2. Si no, fallback léxico: contar matches case-insensitive entre las `trabajo.keywords` y los
   nombres de las áreas del profesor.
3. Empate → orden alfabético por nombre del profesor.

Sin nuevo endpoint dedicado. El ranking más sofisticado (cargas activas, calificaciones, ELO,
diversidad) queda para Camino 2.2 cuando exista el hub.

## Estrategia de testing

- **Backend:**
  - Tests de servicio para `MeTrabajoService.crear/actualizar` (ownership, estado, validaciones).
  - Tests de servicio para `InvitacionOrientacionService.crear/aceptar/rechazar/cancelar`
    cubriendo cada validación.
  - Test de integración (`@SpringBootTest`) cubriendo el happy path:
    estudiante crea → invita profesor → profesor acepta → trabajo queda EN_DESARROLLO con orientador.
  - Test del partial unique index: insertar dos invitaciones PENDIENTE para el mismo trabajo
    debe fallar; insertar una ACEPTADA + una nueva PENDIENTE debe permitirse.
- **Frontend:**
  - Build limpio (`ng build`) sin errores de tipos.
  - Smoke test manual en browser cubriendo: crear trabajo, invitar profesor, aceptar como
    profesor, ver trabajo EN_DESARROLLO como estudiante.

## Orden de implementación

1. Backend: migración Flyway + entidad + repo + DTOs + service.
2. Backend: controllers + tests de service + test de integración happy path.
3. Frontend: servicios + modelos.
4. Frontend: feature `mis-trabajos` (list, crear, detalle, componente InvitarOrientadorForm).
5. Frontend: feature `invitaciones` (recibidas) + sidebar update.
6. Smoke test end-to-end.

## Riesgos y decisiones abiertas

- **Visibilidad de las actividades:** se asume `PARTICIPANTES` para los eventos de
  invitación (estudiante + profesor). Si surge la necesidad de exponerlas a admin se ajusta.
- **Notificaciones al profesor:** este camino no construye un sistema de notificaciones —
  el profesor descubre las invitaciones entrando a `/invitaciones-orientacion`. Notificaciones
  push o email quedan fuera de scope.
- **¿Qué pasa si el profesor borra su cuenta mientras la invitación está PENDIENTE?**
  Se ignora por ahora; la consulta podrá devolver invitaciones huérfanas que el frontend
  filtrará. Hardening posterior si se prioriza.
